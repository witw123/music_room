# P2P 分片分发

最后更新：`2026-04-03`

## 当前设计目标

- 服务端不保存音频文件
- 由房主和已缓存成员共同分发歌曲分片
- 尽量让当前播放曲目优先完成本地缓存

## 分片来源

前端诊断里会看到两类来源：

- `live_upload`
  - 原上传者本地文件直接供片
- `local_cache`
  - 其他成员已经缓存的本地分片继续供片

## 数据对象

当前分片广播和缓存依赖这些信息：

- `trackId`
- `totalChunks`
- `chunkSize`
- `availableChunks`
- `mimeType`
- `fileHash`

客户端会把这些数据落到 IndexedDB 的分片表和 manifest 表中。

## 当前调度原则

- 当前曲目绝对优先
- 启动阶段优先补齐从 `0` 到 `当前播放位置 + 启动窗口` 的连续前缀
- 正常播放时维持前向缓冲窗口
- 暂停时优先把当前曲目补满
- 当前曲目完整后，才会进入后台预取其它曲目

## 本地缓存

当前缓存分为两层：

- 分片缓存
  - 用于渐进式播放、断点恢复和对外供片
- 完整曲目缓存
  - 用于 `full-local` 稳定播放

## 与实时音频的关系

P2P 分片和实时音频不是二选一，而是协同关系：

- 实时音频负责秒开
- 分片负责把成员尽快拉到本地播放
- 一旦本地可播，客户端会尝试从实时流切到本地

## 失败时的表现

如果 DataChannel 没打通：

- 成员拿不到分片
- 渐进式本地播放不会启用
- 只能依赖实时音频

如果 Media 通道也没打通：

- 成员既听不到实时流，也拿不到当前曲目的可播本地数据
- 诊断里通常会看到 `ICE disconnected` 或 `failed`
