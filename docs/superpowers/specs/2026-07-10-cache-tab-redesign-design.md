# 缓存页面重做设计

## 目标

重做房间内的缓存 Tab，使缓存状态更准确、信息更精简、操作更明确，并在桌面端和移动端保持易读易用。保留现有 IndexedDB、P2P 分片下载、缓存组装和服务端曲目注册链路，不修改同步播放与缓存播放策略。

## 页面结构

页面由紧凑概览栏和两个列表区组成：

1. 概览栏展示本机缓存歌曲数、缓存占用空间和正在进行的缓存任务数。
2. “房间歌曲”展示其他成员上传的曲目，提供“全部、下载中、可缓存、已完成”筛选。
3. “本机缓存”展示已完成的本地缓存，提供按歌名、艺术家或上传者搜索。

列表使用紧凑横向行，不使用大块卡片。移动端允许信息和操作自然换行，但进度、状态和按钮不得遮挡曲名。

## 房间歌曲状态

状态优先级固定为：

`已缓存 > 组装中 > 校验失败 > 下载失败 > 已暂停 > 等待来源 > 下载中 > 可缓存`

规则如下：

- 已存在可用完整缓存时始终显示“已缓存”。
- `ready` 任务在缓存库刷新前显示“正在完成”，避免退回“可缓存”。
- 失败状态不得被提供者离线覆盖。
- 下载中按钮不可重复触发，只提供“暂停”。
- 暂停提供“继续”，失败提供“重试”，等待来源时保留不可用的等待状态。
- 未知总分片数显示“等待分片信息”，不显示 `0/0`。
- 仅下载中的新鲜正速率显示速度；暂停、失败、离线和零速率不显示“测速中”。
- 仅有有效缓存时长时显示可播放缓冲。

## 本机缓存交互

- 每行展示歌名、艺术家或来源、时长、文件大小和缓存时间。
- 当前用户已将同一文件加入房间曲库时，按钮显示“已在曲库”并禁用。
- 添加曲库、导出和删除各自维护独立的歌曲级异步状态，执行期间禁用重复操作。
- 删除缓存采用行内二次确认，不使用浏览器确认框。
- 无效时间显示“时间未知”，不显示 `Invalid Date`。

## 信息精简

缓存页不展示以下内部实现字段：

- peer ID
- manifest 来源
- provider 数组与内部连接标识
- pending/requestable 分片计数
- TURN、TCP、candidateType 等传输实现细节
- 原始任务枚举

页面只保留用户完成缓存操作所需的信息：状态、进度、速度、缓冲、可用来源结论和错误原因。

## 数据边界

- 缓存内容与任务状态继续存储在浏览器 IndexedDB。
- 房间曲目继续由现有服务端注册接口维护。
- 页面通过现有 `tracks`、`availabilitySummary`、`cacheLibraryTracks` 和 `manualCacheTasks` 派生展示，不新增后端接口。
- 页面重做不得删除、迁移或改变已有缓存数据格式。

## 错误处理

- 单项操作失败后恢复对应按钮，并沿用全局状态消息显示人类可读错误。
- 页面状态以真实任务状态为主，网络可用性只在没有更高优先级状态时参与判断。
- 操作中的单项状态不得阻塞其他歌曲的独立操作。

## 测试

- 为状态优先级、筛选、进度和格式化逻辑编写单元测试。
- 为重复点击保护、独立操作状态、已在曲库和删除二次确认编写组件测试或可测试的纯状态测试。
- 运行 Web 全量测试、类型检查、lint 和生产构建。
