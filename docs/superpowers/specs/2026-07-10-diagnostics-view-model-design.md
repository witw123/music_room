# 诊断视图模型设计

## 目标

将缓存播放诊断从“直接展示底层遥测字段”改为“先计算可靠状态，再分层展示”。默认层只回答当前是否发声、缓存是否可播、同步是否准确以及当前故障；开发详情保留必要的可读指标，不展示内部标识和原始实现枚举。

## 范围

- 重构成员面板和缓存播放诊断面板的状态展示。
- 新增纯函数诊断视图模型，统一状态、文案、等级和数据新鲜度判断。
- 保留底层诊断 schema 中恢复机制仍可能使用的遥测字段，但从 UI props 和界面中删除无用字段。
- 不改变播放源选择、边缓存播放、PCM 解码、同步时钟或 P2P 传输行为。

## 数据原则

1. 播放源只描述选择了哪条链路，不能单独证明已经发声。
2. PCM 发声必须同时满足：引擎处于可运行状态、AudioContext 正在运行、存在已解码和已调度 PCM、输出已连接，并且没有仍生效的阻塞。
3. 浏览器音频授权等待、主动暂停、PCM 等待分片和解码失败必须是不同状态。
4. 分片公告只证明分片对房间可见，不能证明 PCM 已成功读取。界面使用“已声明完整分片”，不使用“文件已在本地可播”。
5. 速率为零不代表正在传输。超过有效期的速率、RTT 和漂移样本不得参与实时状态结论。
6. 计数为零必须原样显示，不使用其他连接计数回退覆盖。
7. 未采集到的数据展示为“暂无有效样本”，不展示误导性的“未知正常状态”。

## 视图模型

新增独立的纯函数模块，将 `LocalMemberPanelState`、当前曲目分片摘要和有效的 peer diagnostics 转换为以下 UI 模型：

- `audibility`：正在发声、等待音频授权、已暂停、等待 PCM 数据、播放失败、尚未建立。
- `playbackMode`：边缓存无损、完整本地缓存、渐进缓存、未建立。
- `cache`：可见分片数、总分片数、PCM 连续读取分片数、前向缓冲、连续缓冲和健康等级。
- `sync`：正常、偏差较大、严重偏差或无有效样本，并提供有效漂移值。
- `transfer`：有效期内的下载、上传、请求 RTT、超时率和样本年龄。
- `dataLink`：真实打开的 DataChannel 数、直连或中继、网络 RTT、发送队列和可读恢复状态。
- `activeIssue`：当前优先级最高且仍生效的问题，不重复展示历史错误。

所有状态使用明确优先级。播放失败高于等待 PCM，等待音频授权高于普通未就绪，主动暂停不得标记为故障。

## 默认层

默认层只展示：

1. 实际播放状态。
2. 播放模式。
3. 当前曲目可见分片与 PCM 连续读取分片。
4. 前向缓冲和缓冲健康。
5. 同步状态及有效漂移。
6. 当前问题；没有问题时不占用额外区域。

成员卡片中的播放状态与高级诊断摘要必须消费同一个视图模型，避免同一时刻出现互相矛盾的文案。

## 开发详情

开发详情按四组折叠展示：

- 音频与 PCM：AudioContext、输出连接、连续分片、解码段、调度段、缓冲时长、当前阻塞或解码错误。
- 缓存传输：实时下载/上传、样本时间、分片请求 RTT、超时率。
- 同步：时钟偏移、校准 RTT、平均漂移、最大漂移和样本状态。
- 数据链路：DataChannel 状态、直连或中继、网络 RTT、发送队列和人类可读恢复状态。

开发详情不显示 peer ID。远端链路以成员昵称和角色显示；无法映射到成员时显示“房间成员”，不回退展示内部 ID。

## 删除项

从界面和 UI 专用类型中删除：

- playback surface、timeline、connection key 和 peer ID。
- room change、原始 recovery、scheduler 等实现枚举。
- `src`、`srcObject`、`readyState`、paused、muted、volume 的逐项展示。
- 重复的播放源、PCM 状态、阻塞和错误字段。
- 不能产生可靠结论的 ICE 配置说明和内部来源字符串；仅在数据链路详情中保留“直连/中继”。
- 最近事件中的内部 peer ID。

底层 shared schema 不因 UI 精简而盲目删除，以免影响恢复策略；只有确认没有任何运行时消费者的字段才可从采集协议删除。

## 精度修复

- `Data` 摘要严格使用打开的 DataChannel 数，不再以连接 peer 数回退。
- 只有大于零且样本未过期的速率才能判定“正在传输”。
- “完整分片”只表示公告数量达到 manifest 总数，同时单独显示 PCM 连续读取数量。
- active source 为 `lossless-local` 时，必须通过 PCM 输出条件才能显示“正在发声”。
- 历史阻塞在 PCM 已恢复并有已调度输出时不再作为当前故障展示。
- 同步等级使用项目既有漂移容差；没有新鲜样本时不得显示“同步正常”。

## 测试

- 视图模型纯函数测试覆盖发声、授权等待、主动暂停、PCM 无数据、PCM 失败和恢复后清除历史阻塞。
- 覆盖完整分片但 PCM 连续读取为零的准确文案。
- 覆盖速率为零、样本过期和真实传输三种状态。
- 覆盖 DataChannel 数为零时不回退到连接 peer 数。
- 覆盖同步正常、偏差、严重偏差和无有效样本。
- 组件测试验证默认层字段集合、开发详情分组以及内部标识不再渲染。
- 完成后运行 Web 全量测试、typecheck、lint 和生产构建。

## 验收标准

- 默认界面能在一屏内准确回答播放、缓存、同步和故障状态。
- 同一个底层状态不会在成员卡片和高级诊断中产生矛盾结论。
- 页面不出现 peer ID、surface key、timeline key 或其他内部标识。
- “正在发声”“正在传输”“同步正常”均由新鲜且充分的信号支撑。
- 边缓存播放、PCM 和同步运行逻辑不发生行为变化。
