# Music Room Web 双站点改版设计与实施方案

最后更新：`2026-07-13`

文档状态：`提案，等待视觉方向确认`

适用版本：`0.2.8+`

主要范围：`apps/web`、Web 路由和前端部署入口

## 1. 文档目的

本文档定义 Music Room Web 的下一阶段产品结构、信息架构、视觉方向、页面规格、组件边界、响应式行为、实施阶段和验收标准。

改版不是重新设计播放协议，也不是扩展新的音乐内容业务。核心目标是在保留现有房间、同步播放、P2P 分片、本地缓存和诊断能力的前提下，把当前 Web 前端整理为两个职责清晰的站点体验：

- 产品介绍站：解释 Music Room 是什么、如何工作、为什么值得信任，并引导用户进入产品。
- Web 功能站：承载登录、创建或加入房间、同步播放、队列协作、曲库、缓存、成员和诊断等实际操作。

本文档是后续高保真设计、前端实现、测试和验收的共同基线。

## 2. 背景与当前基础

Music Room 当前已经具备完整核心闭环：

1. 用户登录或注册。
2. 创建房间，或通过房间码加入房间。
3. 导入本地音频，注册房间曲库。
4. 将曲目加入共享队列。
5. 房间成员围绕同一条播放时间线进行播放、暂停、切歌和跳转。
6. 新成员通过 P2P data channel 获取音频分片并渐进播放。
7. 浏览器使用 IndexedDB 保存本地缓存。
8. 成员与诊断界面提供连接、缓存、播放源和恢复状态。

当前前端已经存在以下主要入口：

| 路由 | 当前用途 | 结论 |
| --- | --- | --- |
| `/` | 产品介绍页 | 保留并重做内容节奏 |
| `/auth` | 登录和注册 | 保留并重做产品表达 |
| `/app` | 房间大厅 | 作为唯一功能站首页 |
| `/rooms` | 与 `/app` 相同的房间大厅 | 改为重定向到 `/app` |
| `/room/[roomId]` | 房间工作台 | 保留核心结构并重新组织界面 |

现有设计系统已经明确以下方向，应继续作为约束：

- 近黑色背景，而不是大面积灰色。
- 电蓝色作为信号色，而不是大面积色彩铺陈。
- 房间由“播放舞台”和“操作工作区”组成。
- 当前播放状态必须始终是视觉中心。
- 诊断信息需要准确、可信、可读。
- 不引入推荐流、公共音乐发现、排行榜或流媒体曲库。

## 3. 当前体验问题

### 3.1 站点职责重叠

- `/app` 与 `/rooms` 渲染相同页面，增加维护和导航判断成本。
- 产品介绍页、登录页和大厅都在重复解释产品，用户进入功能站后仍然需要越过营销式内容。
- 介绍站和功能站共享相似的页面外壳，但信息密度和任务目标完全不同。

### 3.2 产品表达不一致

- 登录页强调“歌单”和“记录”，但当前版本真正优先的是房间、同步播放、共享队列、本地音频和连接可靠性。
- 大厅的巨大欢迎标题弱化了“创建或加入房间”这一首要任务。
- 介绍页偏向常规技术产品官网，没有充分表现真实的同步听歌体验。

### 3.3 视觉语言出现漂移

- 已有规范要求以黑色和单一电蓝色为主，但大厅出现固定的紫色环境光和多色标题渐变。
- 大量胶囊标签、超大圆角容器和相似玻璃卡片降低了层级区分。
- 手写 SVG 图标风格、尺寸和描边不完全统一。

### 3.4 房间功能层级仍可优化

- 当前四个主标签同时承载队列、曲库、缓存、成员和诊断，普通操作与技术排障混在同一层级。
- 房间顶部身份、成员状态、分享、房间设置和离开操作缺少统一的命令区域。
- 移动端底部播放器占用较高，容易挤压工作区。
- 重连、音频解锁、音源离线、缓冲不足等状态需要更稳定的界面位置。

## 4. 产品目标与非目标

### 4.1 改版目标

- 让新用户在介绍站首屏理解“和朋友同步收听本地音乐”。
- 让已登录用户进入 `/app` 后立即创建或加入房间。
- 让当前歌曲、同步状态和播放控制在房间内始终清晰可见。
- 在不削弱诊断能力的前提下，降低普通用户面对技术信息的认知负担。
- 统一桌面端和移动端的信息架构，而不是维护两套不同产品。
- 保留现有领域逻辑、API、WebSocket、P2P 和播放运行时边界。
- 为后续房间名称、历史房间和更完整邀请能力预留扩展位置。

### 4.2 非目标

- 不建设公共音乐发现和推荐算法。
- 不建设平台托管音频文件的能力。
- 不在本阶段恢复房间内歌单主入口。
- 不引入排行榜、关注关系、动态流、直播打赏或社交反应。
- 不为了视觉改版重写播放同步、P2P 调度或缓存机制。
- 不把真实可交互界面制作成静态图片。

## 5. 双站点总体架构

### 5.1 目标形态

推荐采用“一个 Next.js 工程、两个站点布局、可选两个域名”的结构。

| 站点 | 推荐域名 | 核心职责 |
| --- | --- | --- |
| 产品介绍站 | `www.example.com` | 品牌、产品介绍、工作原理、隐私、开源信息、转化 |
| Web 功能站 | `app.example.com` | 登录、大厅、房间、播放器、设置和诊断 |

两个站点继续由同一个 `apps/web` 构建和发布：

- 共享颜色、字体、图标、按钮和可访问性基础组件。
- 使用不同的 Next.js Layout 管理导航、页面宽度和背景行为。
- 不复制 API client、会话、播放、缓存或 P2P 代码。
- 第一阶段可以继续使用同域路径部署。
- 需要拆分域名时，由 Nginx 将两个 Host 转发到同一个 Web 服务。
- `app.example.com/` 可以在 Nginx 或 Next.js 中重定向到 `/app`。

### 5.2 推荐路由

| 路由 | 站点 | 是否需要登录 | 说明 |
| --- | --- | --- | --- |
| `/` | 介绍站 | 否 | 产品首页 |
| `/how-it-works` | 介绍站 | 否 | 本地文件、信令、P2P 和同步播放说明 |
| `/privacy` | 介绍站 | 否 | 隐私政策 |
| `/terms` | 介绍站 | 否 | 使用条款 |
| `/auth` | 功能站 | 否 | 登录和注册 |
| `/app` | 功能站 | 是 | 房间大厅 |
| `/room/[roomId]` | 功能站 | 是 | 房间工作台 |
| `/settings` | 功能站 | 是 | 账号、本地缓存、音频和浏览器设置 |
| `/rooms` | 兼容入口 | 视目标路由而定 | 永久重定向至 `/app` |

### 5.3 推荐目录结构

```text
apps/web/src/app/
  layout.tsx
  (marketing)/
    layout.tsx
    page.tsx
    how-it-works/page.tsx
    privacy/page.tsx
    terms/page.tsx
  (workspace)/
    layout.tsx
    auth/page.tsx
    app/page.tsx
    room/[roomId]/page.tsx
    settings/page.tsx
  rooms/page.tsx
```

Route Group 只负责布局和组织，不改变公开 URL。

### 5.4 核心用户路径

```text
介绍站首页
  -> 进入网页端
  -> 登录或注册
  -> 房间大厅
       -> 创建公开房间
       -> 创建私密房间
       -> 输入房间码加入
       -> 返回最近房间
       -> 加入开放房间
  -> 房间工作台
       -> 导入本地音乐
       -> 加入共享队列
       -> 开始同步播放
       -> 查看成员、缓存和连接状态
```

## 6. 设计原则

### 6.1 同步状态优先

“大家是否处于同一个播放状态”比装饰性视觉更重要。当前歌曲、播放状态、音源、缓冲和连接恢复必须使用稳定位置表达。

### 6.2 本地优先需要可感知

用户应该在介绍站、导入流程和房间曲库中明确知道：

- 音频文件不会作为平台媒体上传到服务端。
- 服务端保存协作状态和元数据。
- 音频通过成员设备和本地缓存参与播放。

### 6.3 普通操作与技术诊断分层

连接和缓存信息不能被隐藏，但也不应占据所有用户的主要操作路径。主界面提供简洁健康摘要，详细数据进入诊断抽屉或二级视图。

### 6.4 功能站保持工具属性

功能站不是第二个营销页面。避免超大欢迎标题、装饰性指标、重复产品介绍和大面积卡片网格，优先使用列表、工具栏、面板和抽屉。

### 6.5 动效表达系统变化

动效用于播放、同步、切换、连接和反馈，不用于制造持续视觉噪声。

## 7. 视觉系统

### 7.1 色彩

建议在保留现有变量语义的基础上细化：

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `--background` | `#070708` | 功能站主背景 |
| `--background-secondary` | `#0D0D0F` | 播放器、抽屉和强层级区域 |
| `--surface` | `#111214` | 面板和输入区域 |
| `--surface-raised` | `#17181B` | 菜单、模态框 |
| `--surface-hover` | `#1C1D21` | 可交互悬浮态 |
| `--surface-border` | `rgba(255,255,255,0.09)` | 默认边框 |
| `--foreground` | `#F7F7F5` | 主文字 |
| `--foreground-muted` | `#96969F` | 次要文字 |
| `--accent` | `#2F7DFF` | 主操作、播放进度、同步信号 |
| `--accent-hover` | `#4C91FF` | 主操作悬浮态 |
| `--success` | `#46C878` | 已连接、可播放 |
| `--warning` | `#E8B84A` | 缓冲不足、链路降级 |
| `--danger` | `#EF6A6A` | 删除、断开、不可恢复错误 |

色彩规则：

- 不使用固定紫色光球或蓝紫渐变作为页面主题。
- 专辑封面可以生成低透明度环境色，但必须随当前曲目变化，并限制在舞台区域。
- 绿色、黄色和红色只表达真实状态。
- 大面积背景保持中性黑，避免整个界面变成单一蓝色主题。

### 7.2 字体

- UI 主字体继续使用 `Plus Jakarta Sans`，中文使用系统无衬线回退。
- 后续可评估 `Noto Sans SC`，但必须验证构建、加载和中文字重。
- 房间码、时间、延迟、分片、连接状态使用系统等宽字体。
- 功能站控制文字保持 `12px-14px`，避免继承默认 `16px` 后显得松散。
- 只有介绍站产品名和房间当前歌曲可以使用大号展示字体。
- 字距保持 `0`；仅房间码等仪表信息允许有限正字距。

### 7.3 几何与容器

- 普通按钮、输入、列表项、房间条目：`6px-8px` 圆角。
- 模态框和抽屉：`10px-12px` 圆角。
- 唱片、头像、播放按钮等天然圆形元素保持圆形。
- 页面区段不使用浮动卡片包裹。
- 不允许卡片嵌套卡片。
- 仅房间条目、模态框和真正独立的工具面板使用卡片容器。

### 7.4 图标

- 统一使用 Lucide React，补充依赖前先建立图标清单。
- 播放、暂停、上一首和下一首可以使用填充式媒体图标。
- 导入、分享、复制、设置、退出、下载、删除等操作使用统一描边风格。
- 图标按钮必须提供 `title` 或 tooltip，并具备可见焦点态。
- 不继续在页面组件中分散新增手写 SVG。

### 7.5 动效

- 播放时唱片慢速旋转。
- 同步成功使用一次轻微蓝色信号扫过或状态淡入。
- 面板切换使用 `150ms-220ms` 淡入和轻微位移。
- 重连状态允许低频脉冲，不使用快速闪烁。
- 遵循 `prefers-reduced-motion`。

## 8. 产品介绍站规格

### 8.1 站点目标

介绍站只完成四件事：

1. 说明产品是什么。
2. 展示真实使用体验。
3. 建立本地优先与同步可靠性的信任。
4. 引导用户进入功能站。

### 8.2 顶部导航

桌面端：

- 左侧：Music Room 品牌标识。
- 中间：`工作原理`、`核心能力`、`隐私与本地优先`。
- 右侧：GitHub 图标按钮、`进入网页端` 主按钮。

移动端：

- 保留品牌和 `进入网页端`。
- 其他导航进入菜单。
- Header 高度不超过 `60px`。

### 8.3 Hero 首屏

内容要求：

- H1：`Music Room`
- 主说明：`和朋友同步收听你设备里的音乐。`
- 补充说明：`创建房间、导入本地音频，共享同一条播放时间线。文件无需上传到平台媒体库。`
- 主 CTA：`进入网页端`
- 次 CTA：`了解如何工作`

视觉结构：

- 产品名必须是首屏第一信号。
- 不在 H1 上方添加英文 eyebrow、标签或胶囊。
- 真实房间工作台画面占据首屏主要媒体区域。
- 工作台画面需要显示当前歌曲、队列、成员、同步状态和底部播放器。
- 首屏底部露出下一节内容，避免形成封闭满屏海报。
- 不使用与产品无关的音乐人照片或氛围型库存图片。

### 8.4 “三步开始”区段

使用开放式横向流程，而不是三个独立营销卡片：

1. `创建一个房间`
2. `导入本地音乐`
3. `一起同步播放`

每一步配一张真实界面局部或轻量动态图，文字保持操作性。

### 8.5 本地优先区段

核心文案：`音乐留在设备里，协作发生在房间中。`

展示简化的数据流：

```text
本地音频
  -> 浏览器读取与分片
  -> 房间成员之间 P2P 传输
  -> 成员本地缓存与渐进播放

服务端
  -> 账号、房间、队列、播放状态和信令
  -> 不托管房间音频本体
```

### 8.6 同步与可靠性区段

只展示真实能力：

- 共享播放、暂停、切歌和 seek。
- WebSocket 房间状态同步。
- WebRTC P2P 分片传输。
- IndexedDB 本地缓存。
- 断线恢复和播放源诊断。

避免使用未经测量的“零延迟”“全球无缝”等绝对化表述。

### 8.7 浏览器与开源区段

- 说明支持现代桌面和移动浏览器。
- 提示浏览器音频自动播放限制和首次播放解锁。
- 提供 GitHub 仓库、部署文档和问题反馈入口。

### 8.8 Footer

必须提供有效链接：

- GitHub
- 隐私政策
- 使用条款
- 进入网页端

不得保留指向 `#` 的占位链接。

## 9. 功能站通用外壳

### 9.1 Workspace Header

功能站 Header 采用安静、固定的工具栏：

- 左侧：品牌，点击返回 `/app`。
- 中间：根据页面显示当前区域或房间身份。
- 右侧：连接摘要、设置、账号菜单。
- 清除本地缓存进入设置页，不继续作为全局一级按钮。
- 桌面高度建议 `56px`，移动端建议 `52px`。

### 9.2 全局反馈

- 成功反馈：短时 toast，不阻挡主控制。
- 可恢复警告：固定状态条或行内提示。
- 不可恢复错误：明确错误页面或模态框。
- 播放和连接状态使用 `aria-live="polite"`。
- 删除房间、删除缓存和退出房间需要确认。

## 10. 登录与注册页规格

### 10.1 页面目的

只让用户安全、快速地进入功能站，不重复完整产品介绍。

### 10.2 页面结构

- 左上角提供品牌和“返回产品首页”。
- 桌面端可使用低干扰的房间舞台背景或抽象唱片环境。
- 表单是唯一主要容器。
- 登录与注册使用清晰的分段控制或文本切换。

登录文案：

- 标题：`登录 Music Room`
- 说明：`登录后创建房间，或继续加入朋友的房间。`
- 按钮：`登录`

注册文案：

- 标题：`创建账号`
- 说明：`你的昵称会显示在房间成员列表中。`
- 按钮：`创建并进入`

### 10.3 表单状态

- 默认、聚焦、输入完成、提交中、错误、成功。
- 用户名和密码错误统一使用服务端可公开的错误信息。
- 注册时说明昵称用途。
- `redirectTo` 行为继续保留。
- “清除本地数据缓存”移动到故障排查入口或设置页。

## 11. 房间大厅 `/app` 规格

### 11.1 主要任务排序

大厅任务按以下优先级排列：

1. 输入房间码加入。
2. 创建房间。
3. 返回最近房间。
4. 浏览开放房间。

### 11.2 桌面布局

首屏使用工作台布局，不使用营销 Hero：

- 顶部命令区：房间码输入、加入按钮、创建房间按钮。
- 最近房间：一条宽列表项，显示房间码、当前歌曲、在线人数、最后状态和返回操作。
- 开放房间：紧凑列表，显示房主、在线人数、当前播放和加入操作。
- 页面内容宽度建议 `1120px-1280px`。

### 11.3 创建房间弹窗

第一阶段只使用已有后端能力：

- 标题：`创建房间`
- 可见性：公开 / 私密，使用单选或 segmented control。
- 说明公开房间会出现在开放房间列表中。
- 主按钮：`创建并进入`

房间名称、封面和成员权限不在第一阶段表单中伪造。

### 11.4 加入房间

- 房间码输入固定为六位大写等宽字符。
- 支持粘贴后自动去除空格和分隔符。
- 输入完成后可以按 Enter 提交。
- 错误区分无效房间码、房间不存在、房间已结束和网络错误。
- 加入成功后使用现有 snapshot handoff 降低跳转等待。

### 11.5 最近房间

显示：

- 房间码。
- 当前或最近歌曲。
- 在线成员数。
- 自己是房主还是成员。
- `返回房间` 操作。

当前后端只提供最近一个房间，因此第一阶段只显示单条，不伪造完整历史列表。

### 11.6 开放房间列表

每行显示：

- 房主昵称。
- 房间码的弱化展示。
- 在线人数。
- 当前歌曲或“等待播放”。
- 加入按钮。

列表每 10 秒刷新，同时响应页面重新获得焦点和可见性变化，延续现有逻辑。

### 11.7 移动端

- 房间码输入和加入按钮占据第一屏。
- 创建房间按钮与加入操作保持单手可达。
- 最近房间位于开放房间之前。
- 公开房间使用单列行，不使用横向滚动卡片。

## 12. 房间工作台 `/room/[roomId]` 规格

### 12.1 总体结构

桌面端继续保留“舞台 + 工作区 + 固定播放器”：

```text
+-----------------------------------------------------------+
| 房间 Header：返回 / 房间码 / 在线成员 / 分享 / 房间菜单 |
+-------------------------------+---------------------------+
|                               | 工作区 Tabs               |
| 当前播放舞台                  | 队列 / 曲库 / 下载 / 成员 |
| 唱片或封面                    |                           |
| 歌曲与音源                    | 当前 Tab 内容             |
| 同步健康摘要                  |                           |
|                               |                           |
+-------------------------------+---------------------------+
| 连接状态条：同步 / P2P / 缓冲 / 打开诊断                 |
+-----------------------------------------------------------+
| 固定播放器                                                |
+-----------------------------------------------------------+
```

建议桌面比例：

- 舞台约 `58%`。
- 工作区约 `42%`。
- 工作区最小宽度 `420px`，舞台最小宽度 `520px`。
- 小于可用宽度时进入上下堆叠布局。

### 12.2 房间 Header

左侧：

- 返回大厅。
- Music Room 小型品牌标识。

中间：

- 房间码。
- 在线人数。
- 公开或私密状态。

右侧：

- 分享邀请。
- 房间菜单。
- 账号菜单仅在空间允许时显示。

房间菜单包含：

- 复制房间码。
- 离开房间。
- 房主可见的解散房间。

### 12.3 当前播放舞台

必须始终显示：

- 当前曲目视觉：封面优先，无封面时使用统一唱片视觉。
- 歌曲标题和歌手。
- 播放或暂停状态。
- 当前音源提供者。
- 本机播放方式：完整本地缓存或渐进式本地播放。

没有当前曲目时：

- 不显示大型空白唱片占位。
- 显示安静的空舞台和一个明确操作：`从曲库添加音乐`。
- 同时提示也可以把队列中的曲目设为当前播放。

### 12.4 工作区主标签

第一阶段使用四个主标签：

| 标签 | 主要内容 |
| --- | --- |
| `队列` | 共享队列、当前项、排序、播放和移除 |
| `曲库` | 房间曲目、本地导入、加入队列、删除 |
| `下载` | 手动缓存任务、可用源、本地缓存库、导出和删除 |
| `成员` | 在线状态、成员身份、基础传输摘要 |

“缓存”建议在用户界面改称“下载”，技术模型和内部目录可以继续使用 `cache`。

详细 ICE、P2P、分片事件和调度信息进入诊断抽屉，不占据成员页主要区域。

### 12.5 共享队列

- 当前项使用左侧蓝色信号线和清晰的播放状态，不依赖大面积蓝色背景。
- 每行显示序号、歌曲、歌手、时长和添加者。
- 桌面端悬浮后显示操作，移动端通过行尾菜单显示。
- 拖拽排序必须有键盘替代方案，或提供上移/下移菜单操作。
- 队列为空时显示 `从曲库添加第一首歌`。
- “加入队列”不自动开始播放，继续遵循现有产品规则。

### 12.6 曲库与导入

曲库顶部命令区：

- `导入本地音乐` 主操作。
- 曲目数量。
- 可选的本地筛选或搜索在曲目数量明显增长后再启用。

导入过程必须显示：

- 正在读取文件。
- 正在解析元数据。
- 正在计算文件标识或分片。
- 已加入房间曲库。
- 失败原因。

始终显示本地优先说明：`文件保留在你的设备中，房间只同步曲目信息和播放状态。`

### 12.7 下载与本地缓存

下载页分为两个开放区段，不使用嵌套卡片：

- 当前房间曲目的缓存任务。
- 本机缓存库。

每个任务显示：

- 曲目。
- 当前可用源。
- 下载进度。
- 运行、暂停、完成或失败状态。
- 导出、加入曲库或删除操作。

### 12.8 成员

成员行显示：

- 昵称和在线状态。
- 房主身份。
- 是否为当前音源提供者。
- 简洁的传输状态。

本机行允许显示更详细摘要，但不直接展示完整事件流。

### 12.9 连接状态条与诊断抽屉

播放器上方保留一条稳定的技术状态条：

- 同步状态：已同步、校准中、断开。
- Data peers 数量。
- 当前缓冲状态。
- 当前播放方式。
- `查看诊断` 操作。

诊断抽屉包含：

- WebSocket 状态。
- ICE 配置来源与状态。
- Data channel 与 media connection 状态。
- 节点连接详情。
- 分片可用性。
- 最近 P2P 事件。
- 本机上传、下载和缓存命中摘要。

诊断抽屉面向排障，允许高密度等宽信息，但仍需提供面向用户的摘要说明。

### 12.10 底部播放器

桌面端：

- 左侧：唱片或封面、歌曲、歌手。
- 中间：上一首、播放/暂停、下一首。
- 右侧：时间、进度、音量。
- 进度条使用唯一强蓝色信号。

移动端：

- 默认折叠高度约 `64px`，显示歌曲、播放/暂停、下一首和细进度线。
- 点击歌曲区域展开完整播放器 Sheet。
- 展开页包含封面、歌曲信息、完整进度、上一首、播放、下一首、音量和播放方式。
- 适配 `safe-area-inset-bottom`。

### 12.11 房间内关键状态

| 状态 | 表达方式 | 主要操作 |
| --- | --- | --- |
| 初次进入 | 舞台骨架 + 连接摘要 | 等待快照 |
| 需要音频解锁 | 舞台中央明确覆盖层 | 启用音频 |
| 无当前歌曲 | 安静空舞台 | 打开曲库 |
| 当前音源离线 | 固定警告条 | 等待恢复或选择其他曲目 |
| 缓冲不足 | 状态条黄色提示 | 查看下载或诊断 |
| WebSocket 重连 | 顶部非阻塞提示 | 自动重连 |
| 房间被解散 | 独立结束页 | 返回大厅 |
| 房间不存在 | 独立错误页 | 返回大厅或输入房间码 |
| 无权执行操作 | 行内反馈或 toast | 保持当前页面 |

## 13. 邀请与分享面板

从房间 Header 打开，包含：

- 大号等宽房间码。
- 复制房间码。
- 复制房间链接。
- 浏览器原生分享能力可用时显示分享按钮。
- 二维码可以纯前端生成，不需要新增服务端接口。
- 明确公开或私密状态。

第一阶段不新增邀请联系人、社交关系或消息发送能力。

## 14. 设置页 `/settings`

设置页使用全宽分组列表，不使用卡片嵌套。

### 14.1 账号

- 用户名。
- 房间昵称。
- 退出登录。

昵称修改只有在服务端支持后才提供可编辑控件。

### 14.2 音频

- 浏览器音频能力摘要。
- 当前音量。
- 音频自动播放状态。
- 可选输出设备仅在浏览器能力允许时提供。

### 14.3 本地存储

- IndexedDB 缓存使用量。
- 已缓存曲目数。
- 导出或删除单个缓存入口。
- `清除全部本地缓存` 危险操作。

### 14.4 连接诊断

- WebSocket、WebRTC、IndexedDB 和音频解码能力检查。
- 提供复制诊断摘要，但不复制敏感会话信息。

## 15. 响应式规则

### 15.1 推荐断点

沿用 Tailwind 断点语义，并为房间工作台补充容器判断：

- `< 640px`：手机。
- `640px-1023px`：大手机和平板。
- `>= 1024px`：桌面双区布局。
- 短屏高度 `< 760px`：压缩舞台纵向间距，但不缩小主要按钮触控区域。

### 15.2 稳定尺寸

- Header、Tabs、播放器、头像和图标按钮使用稳定高度。
- 队列行、成员行和房间行不得因状态标签出现而跳动。
- 歌曲标题使用截断或最多两行，不得遮挡控制。
- 诊断数据允许横向滚动，但主页面不得产生横向溢出。
- 移动端按钮最小触控尺寸 `44px`。

### 15.3 移动端房间结构

```text
房间 Header
紧凑当前播放舞台
Sticky Tabs
当前 Tab 内容
连接摘要
折叠播放器
```

诊断、分享、播放器详情和房间菜单使用底部 Sheet 或全屏 Dialog。

### 16.2 业务逻辑边界

- `features/playback` 保持播放运行时和同步逻辑。
- `features/p2p` 保持连接、分片和诊断逻辑。
- `features/upload` 保持导入、缓存和任务逻辑。
- `features/session` 保持会话恢复逻辑。
- 页面组件只组合 view model、状态和事件处理。
- 不把 UI 专用 view model 移入 `packages/shared`。
- 不在新组件中复制 API DTO 或 WebSocket payload。

### 16.3 Server 与 Client Component

- 介绍站默认使用 Server Component，减少客户端 JavaScript。
- 登录、大厅、房间和设置中的交互区域使用 Client Component。
- 动态加载曲库、下载、成员和诊断等较重面板。
- 房间运行时保持单一所有者，避免因视觉拆分创建重复 WebSocket 或 P2P 实例。

## 18. 可访问性

- 所有页面保持语义化标题层级。
- Tabs 支持左右方向键、Home、End 和正确的 `aria-selected`。
- 模态框和 Sheet 管理焦点锁定与关闭后焦点恢复。
- 图标按钮具有可访问名称和 tooltip。
- 颜色不是状态的唯一表达方式。
- 正文和操作文字达到 WCAG AA 对比度。
- 键盘可以完成登录、创建房间、输入房间码、队列操作和播放控制。
- 播放和连接提示使用适当的 live region，避免高频播报进度。
- 尊重减少动态效果设置。

## 19. 性能要求

- 介绍站不加载房间运行时、P2P、音频解码器和 IndexedDB 逻辑。
- 功能大厅不加载房间播放运行时。
- 房间面板继续按需动态加载。
- 图片使用稳定宽高比，避免布局偏移。
- 动画优先使用 transform 和 opacity。
- 音频可视化根据设备能力限制 DPR 和刷新频率。
- 不因视觉改版增加重复轮询、WebSocket 订阅或 PeerConnection。

## 22. 发布策略

- 每个阶段保持可独立合并和部署。
- 视觉组件先接入现有事件处理，不同时改业务协议。
- 房间工作台改版可以通过环境变量或内部 feature flag 灰度。
- 第一阶段继续保持同域路径，确认稳定后再启用独立域名。
- 域名拆分前检查 Cookie、CORS、WebSocket、CSP 和反向代理配置。
- 介绍站与功能站应共享版本号，但可以独立观察性能指标。

## 23. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 视觉重构影响播放运行时 | 保持 MusicRoomApp 和运行时所有权，先换外壳再拆组件 |
| 移动端播放器遮挡内容 | 使用固定高度、safe area 和页面底部占位 |
| 诊断信息被过度隐藏 | 保留状态条摘要和一键诊断抽屉 |
| 域名拆分导致鉴权或 WS 问题 | 第一阶段同域，部署前专项验证 |
| 页面概念加入不存在的能力 | 高保真设计只使用现有模型，新增能力单独标记 |
| 大量文件移动增加回归 | 分阶段迁移，不在视觉改版中一次性重命名领域模块 |

## 24. 待确认决策

进入高保真设计前需要确认：

1. 是否接受“一个工程、两个站点布局、后续可拆域名”的架构。
2. 功能站是否使用 `/app` 作为唯一大厅入口。
3. 房间主标签是否采用 `队列 / 曲库 / 下载 / 成员`。
4. 是否接受把详细诊断移入抽屉，同时保留固定健康摘要。
5. 移动端是否采用折叠播放器加完整播放器 Sheet。
6. 第一阶段是否保持现有房间无名称模型。

## 25. 参考方向

以下产品只作为交互原则参考，不作为视觉复制目标：

- Spotify Jam：低门槛加入、共享队列和成员协作。
- Apple SharePlay：同步状态、权限和系统可信感。
- Roon：当前播放舞台、音乐信息层级和沉浸感。
- Linear：安静的工具外壳、命令密度和状态反馈。

最终视觉必须继续符合 Music Room 自身的“本地优先、精确同步、低光技术听音空间”定位。

## 26. 完成定义

本改版完成需要同时满足：

- 产品介绍站和功能站职责清晰。
- `/app` 成为唯一房间大厅入口。
- 介绍站准确表达真实能力，没有无效链接或未经验证的夸张声明。
- 大厅首屏可以直接创建或加入房间。
- 房间内当前歌曲、同步健康和播放控制始终可见。
- 队列、曲库、下载、成员和诊断能力完整保留。
- 桌面与移动端共享一致的信息架构。
- 现有播放同步、P2P、缓存和重连测试无回归。
- 可访问性、响应式和视觉验收达到第 20 节要求。
- 实现结果与最终确认的高保真概念保持一致。
