# 风险与约束

最后更新：`2026-07-15`
当前版本：`0.2.8`

## 网络与基础设施风险

- WebRTC 在复杂 NAT、企业网络、防火墙环境下仍可能建连失败
- TURN 配置不完整时，控制或媒体连接可能进入 `reconnecting` 或 `failed`
- Redis 不可用会直接影响 Realtime、播放控制和跨实例广播
- 宿主机直部署如果错误复用 Docker 版 Nginx upstream，最常见表现就是 `502`

## 产品与体验风险

- 项目不提供公共版权曲库，内容供给依赖用户本地音乐
- 房间使用“曲目拥有者作为媒体源”模型，拥有者弱网或离线会影响其他成员的播放
- 监听端必须满足浏览器 AudioContext 解锁和自动播放策略
- 歌单后端能力保留，但房间主 UI 默认以队列、曲库和成员为主

## 播放运行时风险

- IndexedDB 读取、Opus 解码或浏览器后台限速可能造成短暂 underrun
- RTP 丢包、jitter 或媒体 ICE 重连可能造成缓冲等待
- 多实例和服务重启后的完整状态恢复仍需要更多生产验证
- 旧播放链路和运行时 fallback 已删除，问题只能在当前 Segmented Opus/WebRTC 链路中修复或整体回滚版本

## 数据与部署风险

- 音频原始文件和分段播放资产只保存在用户浏览器；清理浏览器站点数据会删除本地个人曲库
- 生产环境如果误用了认证 fallback 存储，容器重建后会出现账号数据丢失
- 房间删除要求曲目上传者在线，否则当前实现会拒绝删除
- 删除曲目会联动影响队列、播放状态和歌单内容，测试时需要把这类联动当成高风险回归点

## 合规风险

- 平台不托管音频文件，仍需要关注版权风险
- 如果房间被用于非法传播或公开分发，仍然会落到平台责任讨论范围
- 当前更适合把产品定义为“小范围协作听本地音乐工具”，避免开放式音乐分发平台的定位

## 当前建议

现阶段最值得优先压降的风险集中在：

1. Realtime / Redis / TURN 稳态
2. source owner 离线与媒体连接恢复
3. AudioContext 解锁、长时间播放和浏览器后台行为
4. 浏览器级双 context E2E 与真实设备音频观测
