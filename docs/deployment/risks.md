# 风险与约束

最后更新：`2026-07-07`
当前版本：`0.2.8`

## 网络与基础设施风险

- WebRTC 在复杂 NAT、企业网络、防火墙环境下仍可能建连失败
- TURN 配置不完整时，复杂网络下分片同步会退化
- Redis 不可用会直接影响 Realtime、播放控制和跨实例广播
- 宿主机直部署如果错误复用 Docker 版 Nginx upstream，最常见表现就是 `502`

## 产品与体验风险

- 项目不提供公共版权曲库，内容供给依赖用户本地音乐
- 房间体验仍以“房主上传 + 成员协作”模式为主，弱网房主会影响其他成员首播体验
- 歌单后端能力虽然保留，但前端主入口已收起，用户对歌单能力的可发现性偏弱
- FLAC 当前虽然已支持渐进式本地播放，但首次起播仍依赖足够连续的本地缓冲

## 运行时风险

- 房间退出、重连恢复和播放控制仍有偶发竞态，需要继续观察
- 多实例和服务重启后的完整状态恢复还没有形成完全稳态证明
- 浏览器后台限速会影响分片互传、心跳续租和缓存恢复
- 桌面壳与 Android 壳依赖同一套远端入口，打包时如果 `MUSIC_ROOM_PUBLIC_ORIGIN` 配错，会直接影响客户端可用性

## 数据与部署风险

- 生产环境如果误用了认证 fallback 存储，容器重建后会出现“账号像被清空”的现象
- 房间删除要求曲目上传者在线，否则当前实现会拒绝删除
- 删除曲目会联动影响队列、播放状态和歌单内容，测试时需要把这类联动当成高风险回归点

## 合规风险

- 平台不托管音频文件，仍需要关注版权风险
- 如果房间被用于非法传播或公开分发，仍然会落到平台责任讨论范围
- 当前更适合把产品定义为“小范围协作听本地音乐工具”，避免开放式音乐分发平台的定位

## 当前建议

现阶段最值得优先压降的风险集中在：

1. Realtime / Redis / TURN 稳态
2. 重连恢复与多实例行为
3. 部署配置错误导致的 `502` 与客户端入口失效
4. 浏览器级 E2E 回归覆盖不足
